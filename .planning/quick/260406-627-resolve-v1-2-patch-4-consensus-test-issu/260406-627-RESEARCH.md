# Quick Task 260406-627: LanguageModel Prompt API -- Headless vs Headed Browser Modes

**Researched:** 2026-04-06
**Domain:** Browser Built-in AI, Playwright automation, headless mode compatibility
**Confidence:** MEDIUM (no official documentation confirms headless support; findings synthesized from multiple indirect sources)

## Summary

The LanguageModel Prompt API (Browser Built-in AI) has **no officially documented support** for headless browser modes. The API requires GPU inference (5.5 GB VRAM for Edge, >4 GB for Chrome) or CPU inference (Chrome 140+ only, 16 GB RAM + 4 cores). When Playwright uses `channel: 'msedge'`, it launches the **new headless mode** (full browser without display), not the stripped-down `chrome-headless-shell`. This means the full browser feature set is theoretically available, but GPU/WebGPU behavior "will vary" in headless mode per Playwright's own documentation. The model download workflow, component registration, and hardware performance checks create additional barriers for automated contexts.

For the application-dev plugin, the critical implication is: **tests that exercise LanguageModel features may silently get `"unavailable"` from `LanguageModel.availability()` in headless mode**, producing false negatives. The graceful degradation pattern already in the browser-built-in-ai skill handles this correctly (hiding AI features when unavailable), but critics evaluating AI feature quality cannot verify those features actually work unless the API is available.

**Primary recommendation:** Treat headless LanguageModel testing as best-effort. The Generator and Critics should already handle `"unavailable"` gracefully. For the console warning about output language, add `expectedOutputs: [{ type: 'text', languages: ['en'] }]` to `LanguageModel.create()` calls.

## 1. Headless Mode Architecture in Playwright

### Three Modes, Three Behaviors

| Mode | What Runs | When Used | Feature Set |
|------|-----------|-----------|-------------|
| `chrome-headless-shell` | Lightweight content-only binary | Default headless (no channel set) | Minimal -- `//content` module only |
| New Headless (channel) | Full browser, no display window | `channel: 'msedge'`, `channel: 'chrome'`, etc. | Full Chrome/Edge feature set (in theory) |
| Headed | Full browser with visible window | `headless: false` | Full feature set |

**Key finding:** When the application-dev plugin sets `channel: 'msedge'`, Playwright uses the **new headless mode** -- the full Edge browser running without a visible window. This is NOT the stripped-down `chrome-headless-shell`. The new headless mode was designed to be feature-equivalent to headed mode.

**Confidence:** HIGH -- confirmed by Playwright v1.49 release notes and Chromium documentation. Since Chrome 132, `--headless` launches new headless mode (old headless removed from Chrome binary). Playwright v1.57+ uses `chrome-headless-shell` only for default (no-channel) headless.

### GPU/WebGPU in New Headless

Playwright's own documentation for the v1.49 headless changes states: "GPU/WebGL availability, features and performance will vary" between headless shell and new headless mode. This is the most relevant caveat -- the LanguageModel API relies on GPU/CPU inference, and GPU behavior in headless mode is not guaranteed identical to headed mode.

**Confidence:** MEDIUM -- the statement is from Playwright's own issue tracker but lacks specifics about which GPU features are affected.

## 2. LanguageModel API Requirements by Browser

### Chrome (Gemini Nano)

| Requirement | GPU Path | CPU Path (Chrome 140+) |
|-------------|----------|----------------------|
| VRAM | >4 GB | Not needed |
| RAM | -- | >=16 GB |
| CPU Cores | -- | >=4 |
| Storage | >=22 GB free | >=22 GB free |
| OS | Windows 10/11, macOS 13+, Linux, ChromeOS | Same |

CPU inference rolling out in Chrome 140. No code changes needed -- same API, potentially slower inference.

### Edge (Phi-4-mini)

| Requirement | Value |
|-------------|-------|
| VRAM | >=5.5 GB |
| Storage | >=20 GB free |
| OS | Windows 10/11, macOS 13.3+ |
| Channel | Edge 138+ Canary/Dev |
| CPU fallback | **Not documented** for Edge built-in API |

Edge documentation only lists GPU VRAM as a requirement. Unlike Chrome, **no CPU fallback is documented for Edge's built-in Phi-4-mini**. This means the user's Qualcomm Adreno X1-85 GPU with limited VRAM may not meet Edge's 5.5 GB VRAM requirement.

**Confidence:** HIGH -- sourced from Microsoft's official Edge documentation (learn.microsoft.com, updated 2026-02-04).

### Hardware Check: Device Performance Class

Both browsers expose an internals page to check compatibility:
- Chrome: `chrome://on-device-internals`
- Edge: `edge://on-device-internals`

Edge requires "Device performance class" of **High or greater**. This check may fail in headless mode where GPU enumeration differs from headed mode.

## 3. Enabling Built-in AI in Automated Contexts

### Chrome Command-Line Flags

```
--enable-features=OptimizationGuideOnDeviceModel:compatible_on_device_performance_classes/*,PromptAPIForGeminiNano
```

The `compatible_on_device_performance_classes/*` parameter is the command-line equivalent of the `BypassPerfRequirement` dropdown in `chrome://flags`. This bypasses the hardware performance check.

### Edge Command-Line Flags

Edge flag name in `edge://flags`: **"Prompt API for Phi mini"**

Probable command-line equivalent (not officially documented):
```
--enable-features=PromptAPIForPhiMini
```

Additional Edge-specific flags found in community sources:
- `enable-phi-mini` -- activates core Phi-4-mini model
- `enable-phi-mini-prompt-api` -- enables Prompt API specifically
- `on-device-model-performance-override` -- bypasses performance checks

**Confidence:** LOW for Edge command-line flags. The exact PascalCase feature names for `--enable-features` are not officially documented by Microsoft. The flag names above are inferred from `edge://flags` kebab-case naming conventions.

### Automation Barriers

| Barrier | Chrome | Edge |
|---------|--------|------|
| Component updates disabled | Puppeteer/Playwright disable `--disable-component-update` by default | Same (Chromium-based) |
| Model download | Requires component update + time | Same |
| User activation for download | Required if model status is `"downloadable"` | Same |
| Profile state | Model tied to browser profile | Same |
| Incognito/Guest | Not supported | Not documented |

**Workaround for component updates in Playwright:**

In Puppeteer, `ignoreDefaultArgs: ['--disable-component-update']` is needed. Playwright equivalent:

```javascript
const browser = await chromium.launch({
  channel: 'msedge',
  ignoreDefaultArgs: ['--disable-component-update'],
  args: [
    '--enable-features=PromptAPIForPhiMini'
  ],
});
```

**Confidence:** MEDIUM -- the Puppeteer approach is documented in issue #13011. Playwright's `ignoreDefaultArgs` follows the same pattern but has not been explicitly verified for this use case.

### Pre-Downloaded Model Approach

The most reliable automation path is to use a browser profile that already has the model downloaded:

1. Manually open Edge, enable flags, wait for model download
2. Point Playwright at that user data directory via `--user-data-dir`
3. The model persists across sessions

This avoids the component-update and user-activation barriers entirely.

## 4. Implications for Application-Dev Plugin

### Current Architecture

- **Generator**: Uses `npx playwright-cli screenshot --browser msedge` (headless by default)
- **Critics**: Use `npx playwright test` with `channel: 'msedge'` in test config (headless by default)
- **Generated apps**: May include LanguageModel API features via browser-built-in-ai skill

### Impact Assessment

| Component | Impact | Severity |
|-----------|--------|----------|
| Generator screenshots | None -- screenshots don't test AI runtime | None |
| Generator Playwright tests | Tests exercising LanguageModel may get `"unavailable"` | Medium |
| Perceptual Critic | Evaluates visual design, not AI functionality directly | Low |
| Projection Critic | Tests navigation and functionality; AI features behind graceful degradation would be hidden | Medium |
| Perturbation Critic | Tests robustness; graceful degradation should handle unavailable AI gracefully | Low |

### Why This Is Not a Blocker

The browser-built-in-ai skill already mandates **graceful degradation**: when `LanguageModel.availability()` returns `"unavailable"`, the AI feature is hidden or disabled. This means:

1. In headless mode, if the API is unavailable, the app correctly hides AI features
2. Critics see a working app without AI features -- they evaluate what's visible
3. The app doesn't crash or show errors -- the degradation is silent

The real loss is that **AI feature quality cannot be validated** in headless mode if the API is unavailable. The Projection Critic cannot verify that the chatbot actually responds, the Summarizer actually summarizes, etc. These features would appear as gracefully degraded (hidden) rather than functional.

### Recommended Approach

**Do not attempt to force headless LanguageModel support.** Instead:

1. Accept that AI features are best-effort in automated testing
2. Ensure the graceful degradation pattern is robust (already is)
3. If AI feature validation is critical, document that headed mode or a pre-configured profile is needed
4. The console warning about output language is a code quality issue, not a headless issue -- fix it regardless

## 5. Output Language Warning -- How to Fix

### The Warning

```
No output language was specified in a LanguageModel API request.
An output language should be specified to ensure optimal output quality
and properly attest to output safety.
Please specify a supported output language code: [en, es, ja]
```

### The Fix

Add `expectedOutputs` (and optionally `expectedInputs`) to `LanguageModel.create()`:

```javascript
const session = await LanguageModel.create({
  initialPrompts: [
    { role: 'system', content: 'You are a helpful assistant.' }
  ],
  expectedInputs: [{ type: 'text', languages: ['en'] }],
  expectedOutputs: [{ type: 'text', languages: ['en'] }],
});
```

### Important Caveats

- `expectedOutputs` does **not** force the model to output in that language -- it only declares an expectation
- It helps the browser set up the session optimally and download supporting material
- It should be passed to both `LanguageModel.availability()` and `LanguageModel.create()` with the same options
- Currently supported output languages: `en`, `es`, `ja`

### Where to Fix in the Plugin

The `references/prompt-api.md` file already shows `expectedOutputs` in its session creation example (line 49). The fix is to ensure the Generator actually follows this pattern. The warning likely comes from test runs where the generated app code called `LanguageModel.create()` without `expectedOutputs`.

**Action:** Update `references/prompt-api.md` to emphasize that `expectedOutputs` is required (not optional) with a warning callout, and consider adding it to the `tryCreateSession` helper in `references/graceful-degradation.md`.

**Confidence:** HIGH -- sourced from W3C spec README and Chrome developer group discussion.

## 6. Chrome vs Edge: Which Channel for Testing?

| Factor | Chrome (`channel: 'chrome'`) | Edge (`channel: 'msedge'`) |
|--------|------------------------------|---------------------------|
| Model | Gemini Nano | Phi-4-mini (3.8B, more capable) |
| CPU fallback | Yes (Chrome 140+, 16GB+4 cores) | Not documented |
| GPU requirement | >4 GB VRAM | >=5.5 GB VRAM |
| API maturity | OT 139-144, shipping 148 | Dev preview 138+ |
| Automation docs | Puppeteer issue #13011 | None found |

For the application-dev plugin, `msedge` remains the right default (Phi-4-mini is more capable than Gemini Nano). The lack of CPU fallback in Edge is a concern for devices without sufficient GPU VRAM, but the graceful degradation pattern handles this.

## Common Pitfalls

### Pitfall 1: Assuming Headless Equals Headed for AI APIs
**What goes wrong:** Tests pass in headed mode (AI features work) but silently degrade in headless (AI features hidden), giving a false sense of quality.
**How to avoid:** Accept this limitation. Document that AI feature validation requires headed mode or pre-configured profile. Do not treat headless test results as comprehensive for AI features.

### Pitfall 2: Missing `expectedOutputs` Causes Console Warnings
**What goes wrong:** The console fills with "No output language was specified" warnings, which may appear in critic evaluation reports as issues.
**How to avoid:** Always pass `expectedOutputs: [{ type: 'text', languages: ['en'] }]` to `LanguageModel.create()`. Update reference docs to make this mandatory, not optional.

### Pitfall 3: Component Updates Disabled by Default
**What goes wrong:** Playwright/Puppeteer disable Chrome component updates by default (`--disable-component-update`). The on-device model is a component. Without it, `LanguageModel.availability()` always returns `"unavailable"`.
**How to avoid:** If attempting AI testing in automation, add `ignoreDefaultArgs: ['--disable-component-update']` and use a pre-downloaded profile.

### Pitfall 4: Edge GPU VRAM Requirement on ARM Devices
**What goes wrong:** The user's Qualcomm Adreno X1-85 GPU may not have 5.5 GB dedicated VRAM. Edge's built-in AI check may classify the device below "High" performance class.
**How to avoid:** Check `edge://on-device-internals` manually to verify device performance class. If it's below "High", Edge's built-in AI will not activate regardless of flags.
