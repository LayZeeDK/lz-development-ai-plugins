### 1. Architecture section

This replaces the originally-planned four-layer enforcement -- plugin hooks were dropped because they are session-wide and cannot distinguish agents."

### 2. Enforcement Model section
Document the two-layer enforcement model:
- Layer 1: Tool allowlists (agent `tools` frontmatter = strict; skill `allowed-tools` = pre-approval, effectively strict in autonomous flow)
- Layer 2: Prompt guards (output-domain constraints in each agent's instructions)
- Note: `disallowedTools` is not available on skills; hooks dropped (session-wide scope)
- This addresses ORCH-05 (revised) and ORCH-06 (revised)

### 3. Rules section
Rewrite with these rules (ORCH-01 enforcement):
1. "Write is ONLY for .appdev-state.json. Never write source code, specs, QA artifacts, or any other files."
2. "Never diagnose agent output beyond binary file-exists checks. Do not read agent output to assess quality, completeness, or correctness."
3. "Never add corrective instructions to agent prompts on retry. Use the exact same prompt for retries as the original spawn."
4. "Never perform agent work. If an agent fails and retries are exhausted, ask the user -- do not attempt the agent's task yourself."
5. "Fully autonomous after initial prompt. Do not ask the user for feedback during the workflow except for error recovery (retry exhaustion)."

### 4. Workflow section
Rewrite the workflow with these steps:

**Step 0: Resume Check**
- Run `Bash(node ${CLAUDE_PLUGIN_ROOT}/scripts/appdev-state.cjs exists)` to check for existing state
- If `{"exists": true}`: Read state with `get`, show the original prompt and completed steps to the user, use AskUserQuestion with options: "Resume from [step]" or "Start fresh (deletes previous progress)"
  - Resume: skip to the appropriate step based on state
  - Start fresh: run `delete` then proceed to Step 1
- If `{"exists": false}`: proceed to Step 1

**Step 1: Plan**
- Output: `[1/3] Planning...`
- Run `Bash(node ${CLAUDE_PLUGIN_ROOT}/scripts/appdev-state.cjs init --prompt "<user's prompt>")` (only if not resuming)
- Spawn Planner: `Agent(subagent_type: "application-dev:planner", prompt: "<user's full prompt, verbatim>")`
- Apply the error recovery pattern (see Error Recovery section below)
- Binary check: Read `SPEC.md` -- verify file exists and contains "## Features". Do NOT assess spec quality (the Planner self-verifies).
- If check fails: retry with same prompt (counts toward the 2-retry limit)
- Update state: `Bash(node ${CLAUDE_PLUGIN_ROOT}/scripts/appdev-state.cjs update --step generate --round 1)`
- Output: `[1/3] Planning... done`

**Step 2: Build/QA Loop**
- Run up to 3 rounds (Phase 2 changes this to score-based exit with 10-round cap).

For each round N:

Build phase:
- Output: `[2/3] Generating (round N)...`
- Update state: `Bash(node ${CLAUDE_PLUGIN_ROOT}/scripts/appdev-state.cjs update --step generate --round N)`
- Round 1 prompt: `"Build the application defined in SPEC.md. This is build round 1 -- there is no prior QA feedback."`
- Round 2+ prompt: `"This is build round N. Read QA-REPORT.md for the Evaluator's feedback from the previous round. Fix the issues found and improve the application."`
- Spawn Generator with the appropriate prompt. Apply error recovery pattern.
- Binary check: Use Bash to check a key project file exists (e.g., `ls package.json index.html pyproject.toml 2>/dev/null`). The Generator self-tests; orchestrator just confirms something was built.
- Output: `[2/3] Generating (round N)... done`

Evaluate phase:
- Output: `[2/3] Evaluating (round N)...`
- Update state: `Bash(node ${CLAUDE_PLUGIN_ROOT}/scripts/appdev-state.cjs update --step evaluate --round N)`
- Prompt: `"Evaluate the application against SPEC.md. This is QA round N. Write your report to QA-REPORT.md."`
- Spawn Evaluator. Apply error recovery pattern.
- Binary check: Read `QA-REPORT.md` -- verify file exists and contains "## Verdict". Do NOT assess report quality (the Evaluator self-verifies).
- If check fails: retry with same prompt
- Parse verdict: search for "PASS" or "FAIL" in the Verdict line. This is the ONLY qualitative read the orchestrator does -- a single keyword match.
- Record round: `Bash(node ${CLAUDE_PLUGIN_ROOT}/scripts/appdev-state.cjs round-complete --round N --verdict <PASS|FAIL>)`
  - Note: scores extraction is deferred to Phase 2. For now, round-complete records only the verdict.
- Output: `[2/3] Evaluating (round N)... Verdict: <PASS|FAIL>`
- If PASS or round = 3: exit loop
- If FAIL and round < 3: next round

**Step 3: Summary**
- Output: `[3/3] Summarizing...`
- Update state: `Bash(node ${CLAUDE_PLUGIN_ROOT}/scripts/appdev-state.cjs update --step summary)`
- Read `QA-REPORT.md` and `README.md` to present a summary to the user:
  - Product name and what was built
  - Key features implemented (from QA report's feature status table)
  - Final QA scores and verdict
  - Number of rounds completed
  - How to start/use the app (from README)
- Complete: `Bash(node ${CLAUDE_PLUGIN_ROOT}/scripts/appdev-state.cjs complete --exit-condition PASS)` (or appropriate condition)
- Delete state: `Bash(node ${CLAUDE_PLUGIN_ROOT}/scripts/appdev-state.cjs delete)`
- Output: `[3/3] Summarizing... done`

Note: The Summary step is the ONE exception where the orchestrator reads agent output in detail. This is presentation to the user, not diagnosis or correction. The orchestrator does not act on this information -- it only formats it for display.

### 5. Error Recovery section
Document the error recovery pattern used by all agent spawns (ORCH-02):

1. Spawn agent with the defined prompt
2. If agent spawn fails (Agent tool returns error):
   - Retry 1: Spawn again with the EXACT same prompt (no modifications, no diagnostic context added)
   - Retry 2: Spawn again with the EXACT same prompt
   - After 2 retries exhausted: Use AskUserQuestion to present options:
     - "Retry now" -- spawns the agent again (unlimited user-initiated retries)
     - "Resume later" -- writes current state and stops (user can run `/application-dev` to resume)
     - "Abort" -- deletes state file and stops
3. Agent tool success/failure status may be unreliable (classifyHandoffIfNeeded bug). Always perform the binary file-exists check regardless of reported status. If expected files exist, treat agent as successful even if Agent tool reported failure.

### 6. Agent Prompt Protocol section (ORCH-03)
Document the exact prompts passed to each agent. The orchestrator MUST use these exact templates -- no additions, no context injection, no failure diagnostics:

- Planner: `"<user's full prompt, verbatim>"` -- nothing else
- Generator (round 1): `"Build the application defined in SPEC.md. This is build round 1 -- there is no prior QA feedback."`
- Generator (round 2+): `"This is build round N. Read QA-REPORT.md for the Evaluator's feedback from the previous round. Fix the issues found and improve the application."`
- Evaluator: `"Evaluate the application against SPEC.md. This is QA round N. Write your report to QA-REPORT.md."`

"The orchestrator fills in only the round number. No free-form additions. No error context. No diagnostic notes. No 'this time make sure to...' instructions."

### 7. File-Based Communication section
Document that agents communicate through `SPEC.md` and `QA-REPORT.md` only. The orchestrator coordinates through `.appdev-state.json` only. No other inter-agent communication paths exist.
