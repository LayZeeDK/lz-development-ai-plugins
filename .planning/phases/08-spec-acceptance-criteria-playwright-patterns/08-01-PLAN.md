  <action>
Modify SPEC-TEMPLATE.md:

1. In the feature template section (between User Stories and Data Model), add an `**Acceptance Criteria:**` section with a comment block containing 2-3 good and 2-3 bad inline examples. The comment block teaches by demonstration.

Good example bullets should demonstrate:
- Observable outcome with natural numeric threshold ("Gallery displays at least 12 artworks on initial load")
- User action with observable result ("User can filter artworks by artist name")
- Persistence stated as user-facing ("Created artwork persists after page refresh")

Bad example bullets should demonstrate:
- Vague quality ("The gallery looks good on mobile")
- UI-prescriptive ("Clicking Add opens a modal with a form")
- Implementation detail ("Data is saved to localStorage")

After the comment block, add 3 placeholder bullets:
```
- <observable outcome with measurable threshold if natural>
- <observable outcome>
- <observable outcome>
```

Keep the rest of the template unchanged. The Acceptance Criteria section goes between User Stories and Data Model.

2. Create NEW file acceptance-criteria-guide.md (~50 lines) in the references/ directory. Structure:

Section 1 - "What makes criteria testable" (~5 lines): Observable outcome from user perspective, one assertion per bullet, include natural thresholds when the spec supplies numbers, do not force synthetic thresholds on binary behaviors.

Section 2 - "Good vs bad examples" (~15 lines): 4-5 universal paired examples covering display counts, user actions, error states, persistence, and responsive behavior. Each pair shows a bad version and its good replacement with a brief WHY.

Section 3 - "Tier rules" (~10 lines): Core features >= 3 criteria with prescribed types (happy path + edge case + error state). Important features >= 2 criteria (count only, types flexible). Nice-to-have features >= 1 criterion (count only).

Section 4 - "Common pitfalls" (~10 lines): Vague qualities, UI-prescriptive language, implementation mechanism leaks, cross-feature criteria. Each pitfall in 1-2 sentences.

Section 5 - "AI feature criteria" (~5 lines): AI outputs are variable -- test behavioral quality not exact content. 2 good/bad pairs specific to AI features.

The guide is consumer-agnostic: it says "criteria must be verifiable by automated tests" without naming the projection-critic, Playwright, or write-and-run. This preserves GAN separation -- the planner should not know evaluation internals.

Use WHY-based rationale (not ALL-CAPS emphasis) consistent with project conventions.
