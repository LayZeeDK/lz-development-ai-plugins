**Structure the file with these sections:**

1. **Purpose statement** -- This file is loaded at Step 12 (Read Calibration + Score) of the evaluation workflow. It anchors scores to concrete scenarios and enforces mechanical ceiling rules.

2. **Hard Score Ceiling Rules** -- Copy EXACTLY from CONTEXT.md decisions. These are mechanical, not judgment:

   Functionality:
   - Any Critical bug -> max 5
   - 3+ Major bugs -> max 6
   - Core workflow broken -> max 4

   Product Depth:
   - >50% features Missing/Broken -> max 5
   - Any Core feature missing -> max 6
   - All features stubbed -> max 3
   - Canned AI feature -> max 5

   Visual Design:
   - All images placeholder -> max 3
   - No design language match -> max 5
   - Layout broken on mobile -> max 5

   Code Quality:
   - Security vulnerability -> max 4
   - No error handling anywhere -> max 5
   - Dead code >30% of codebase -> max 5

   Browser AI Degradation:
   - App non-functional without browser AI APIs -> Functionality max 4
   - AI features show broken UI without APIs -> Functionality max 6

   When multiple ceilings apply, use the LOWEST ceiling.

3. **Conflict Resolution Rules** -- From CONTEXT.md:
   - Criteria are independent (great code + broken features: Code Quality 7, Functionality 3)
   - Cross-criterion propagation applies in one direction: a bug that ALSO affects design lowers BOTH scores. But good code does not RAISE functionality.
   - No averaging or trading between criteria. FAIL if ANY below threshold.

4. **Score-Against-the-Spec Rule** -- The spec is the contract. Beautiful wrong-theme design = low Visual Design. Extra features not in spec do not count for Product Depth.

5. **Round-Independent Scoring** -- Absolute standards, no grading drift. A 5 in round 1 means the same as a 5 in round 5. Do not compare to the previous round -- compare to the spec.

6. **Mandatory Score Justifications** -- Each score in the Scores table must cite specific findings. Format example: "Functionality: 5/10 -- 2 Critical bugs (#3, #7), 3 Major bugs, core search broken."

7. **Calibration Scenarios** -- 3 scenarios per criterion (below threshold, at threshold, above threshold) = 12 total. Each scenario must have:
   - A concrete description (~50 words) of an application state
   - The score and rationale
   - A "not X because" boundary explanation distinguishing it from adjacent scores

   Use Claude's discretion for the specific scenario content, but follow these constraints:
   - Scenarios must reference realistic application states (not abstract descriptions)
   - Below-threshold scenarios should be clearly distinguishable from at-threshold ones
   - Above-threshold scenarios should represent genuine excellence, not just "no bugs"
   - Reference the rubric descriptors from evaluator.md (1-3, 4-6, 7-8, 9-10 ranges)

   Product Depth scenarios: below (5/10), at (7/10), above (9/10)
   Functionality scenarios: below (5/10), at (7/10), above (9/10)
   Visual Design scenarios: below (4/10), at (6/10), above (8/10)
   Code Quality scenarios: below (4/10), at (6/10), above (8/10)

Create AI-PROBING-REFERENCE.md in the references/ directory. This file is loaded at Step 8 (AI Feature Probing) of the evaluation workflow. It describes probe STRATEGIES organized by modality.

**CRITICAL: Goodhart's Law protection.** This file describes probe strategies, NOT fixed scripts with exact inputs. The Evaluator generates domain-appropriate probe inputs on the fly from SPEC.md context. This prevents the Generator from pre-computing responses to known probes.

**Structure the file with these sections:**

1. **Purpose and Philosophy**
   - This file is the Evaluator's adversarial probe reference for AI features
   - Chinese Room foundation: "Probes test semantics (does the AI understand the input?) not syntax (does the response look well-formed?). A grammatically correct response is not evidence of real AI -- the Generator can produce syntactically perfect canned responses."
   - Strategies describe WHAT to test and WHY, not exact inputs to use
   - The Evaluator derives specific inputs from SPEC.md domain context

2. **Three-Tier Detection Model**
   - Tier 1: Behavioral probes (primary evidence) -- what is described in this file
   - Tier 2: Technical signals (supporting evidence) -- latency patterns, network requests to AI APIs, browser API presence checks (window.ai, navigator.ml)
   - Tier 3: NO code scanning. GAN principle: discriminator evaluates output, not generator's internals. Code Quality Assessment is separate.

3. **Probe Ordering Principle** -- "Start with variability and domain probes. Escalate to adversarial probes (nonsense, negation, Theory of Mind). Prevents habituation from masking baseline failures." The 10-probe battery (0-9) is already ordered baseline-to-adversarial.

4. **Universal 10-Probe Battery for Conversational AI Features** -- All 10 probes from CONTEXT.md with:
   - Probe number and name
   - Purpose (what it tests)
   - Strategy (how to generate domain-appropriate input from SPEC.md -- NOT a fixed input)
   - Canned detection signal (what response pattern indicates canned)
   - Real AI signal (what response pattern indicates genuine inference)
   - For Probe 0 (Variability): note the fast exit -- 3 identical responses = definitive canned, skip remaining probes for this feature

5. **Turing Test Concept Probes** -- Each of these is an ADDITIONAL probe technique that can be applied within or after the 10-probe battery:

   a. **Winograd Schema probes** -- Generate domain-appropriate ambiguity sentences from SPEC.md context. Describe the schema pattern (pronoun reference disambiguation), give the strategy for generating domain-specific schemas, and explain why keyword matchers cannot resolve them. Note: LLMs solve these easily; canned systems cannot.

   b. **Grice's Specificity (Quantity Maxim)** -- Ask narrow factual questions where a direct answer is expected. Check whether the response is proportional to the question (real AI) or gives broad keyword-triggered paragraphs (canned). Provide strategy for deriving narrow questions from SPEC.md content.

   c. **Compression Round-Trip** -- Give domain content from SPEC.md, ask to summarize in one sentence, then expand back. Compare expansion to original for information preservation. Canned systems cannot compress-decompress faithfully.

   d. **Complexity Scaling** -- Give inputs of increasing complexity, verify response complexity scales proportionally. Canned systems give similar-length responses regardless of input complexity.

   e. **Theory of Mind** -- Present a scenario with a false belief derived from SPEC.md context. Real AI engages with the user's mental model. Canned systems keyword-match and give generic responses. This is the most advanced probe -- run last.

   f. **Visual Turing Test** -- For Text->Image features. Decompose text prompt into individual elements, ask binary questions via Claude vision: "Does this image contain [element]?" Tally hit rate. Pre-stored stock photo matches few prompt elements; real generation matches most.

6. **Modality-Based Probe Batteries** -- Organized by input/output modality as specified in CONTEXT.md. For each modality, describe:
   - What this modality covers (with examples)
   - Key probe strategies (what to test)
   - Canned detection signals specific to this modality
   - Special considerations

   Modalities from CONTEXT.md:
   - Text -> Text (10-probe battery + Turing test concept probes above)
   - Text -> Image (Visual Turing Test binary questions, nonsense prompts, prompt element verification)
   - Image -> Text (canvas-generated test images, graduated difficulty)
   - Image -> Image (input sensitivity, vision comparison)
   - Text -> Audio (duration correlation, unusual input, API mocking detection)
   - Audio -> Text (fake mic input via Chromium flags, varied audio files)
   - Text -> Structured Data (input sensitivity, edge cases, novel inputs)
   - Interactive / Real-time (behavioral adaptation over time, context sensitivity)
   - Data -> Data (input-output correlation, edge cases)
   - Special: Server-side-only (observe effects only)
   - Special: Invisible AI (cross-session comparison)
   - Special: Game AI (behavioral adaptation over time)

   Use Claude's discretion for detailed content per modality section. The Text->Text section should be the most detailed since it uses the full 10-probe battery + all Turing test concepts. Other modalities should adapt relevant probes to their specific I/O characteristics.

7. **Off-Spec Features Scoring** -- From CONTEXT.md decisions:
   - Product Depth: penalized (off-spec outputs, misallocated effort, GAN precision)
   - Code Quality: penalized (YAGNI violation, added complexity without spec justification)
   - Functionality: bugs from off-spec features count normally
   - Canned off-spec AI: Major (deceptive) + Product Depth + Code Quality hit
   - Feature count decrease between rounds from removing off-spec features is NOT a regression
   - Terminology: "off-spec features" (not "bonus features" or "scope creep")

8. **Canned AI Hard Ceiling** -- Canned AI feature = Product Depth max 5 (deceptive, worse than missing). Cross-criterion: also Major bug in Functionality + Code Quality penalty.

9. **AI Quality Assessment (Beyond Detection)** -- If the AI is confirmed real, also assess: accuracy, relevance, coherence, helpfulness. Real but poor quality -> lower Functionality based on severity.

10. **AI Latency Assessment** -- Feedback-based, not absolute thresholds:
    - Missing loading indicator during inference: Major
    - Missing progress during model download: Critical
    - Missing streaming for long text responses: Minor
    - Frozen UI during inference: Major

11. **Graceful Degradation Hard Rule** -- From CONTEXT.md:
    - LanguageModel is origin trial, not standard. No Firefox, no Safari.
    - Same applies to WebLLM (WebGPU), WebNN (experimental)
    - App non-functional without browser AI APIs: Functionality max 4
    - AI features show broken UI without APIs: Functionality max 6
    - Graceful degradation with clear messaging: no ceiling (progressive enhancement)

12. **Subject Matter Expert Judge** -- The Evaluator reads SPEC.md before AI probing (step 1 -> step 8). By the time it probes AI features, it knows the app's domain. Domain questions and Winograd schemas should reference SPEC.md content, not generic examples.

**Use research/ai-feature-taxonomy.md** to inform modality section structure. The taxonomy organizes AI features by modality which maps directly to the probe battery organization.

**Do NOT include:**
- Fixed probe scripts with exact inputs (Goodhart's Law violation)
- Code scanning instructions (Tier 3 is explicitly NO code scanning)
- Generic examples that do not require SPEC.md context
