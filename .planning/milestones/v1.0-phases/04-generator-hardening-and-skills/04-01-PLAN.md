Create 3 reference files in `plugins/application-dev/skills/playwright-testing/references/`. Each file teaches one phase of the Playwright Test Agents plan/generate/heal workflow.

**test-planning.md** (~150-200 lines):
1. How to explore the running app using `npx playwright-cli` to understand current UI state
2. How to read SPEC.md and identify key user flows worth testing (prioritize: critical paths first, then edge cases)
3. Test plan format: Markdown document in specs/ directory with:
   - Flow name and description
   - Preconditions (app state, data requirements)
   - Steps (numbered user actions)
   - Expected outcomes (observable assertions)
4. Coverage priority: core user flows first, then feature-specific flows, then edge cases
5. Example test plan structure (showing a realistic flow like "guest checkout" or "create new document")
6. When to skip planning and write tests directly (simple apps, obvious flows)

**test-generation.md** (~150-250 lines):
1. How to write Playwright tests from test plans
2. seed.spec.ts pattern: environment setup test that verifies the app is running and accessible (base URL responds, no console errors on load)
3. Test file organization: one file per flow or feature area, tests/ directory mirrors specs/
4. Selector strategy: accessibility-tree-first approach:
   - Prefer: `getByRole()`, `getByLabel()`, `getByText()`, `getByPlaceholder()`
   - Avoid: CSS selectors, XPath, data-testid (unless no semantic alternative)
5. Assertion patterns: `expect(page).toHaveTitle()`, `expect(locator).toBeVisible()`, `expect(locator).toHaveText()`
6. Common patterns:
   - Navigation and route testing
   - Form submission flows
   - Dynamic content loading (wait for network idle or specific element)
   - Screenshot comparison for visual regression (when appropriate)
7. Anti-patterns to avoid:
   - Hard-coded waits (use `waitForSelector` or auto-waiting)
   - Testing implementation details instead of user-visible behavior
   - Overly brittle selectors that break on text changes

**test-healing.md** (~150-200 lines):
1. The healer loop: run tests -> collect failures -> diagnose -> fix -> re-run
2. How to run tests: `npx playwright test` with useful flags (`--reporter=line`, `--workers=1` for debugging)
3. Reading Playwright error output: understanding assertion errors, timeout errors, navigation errors
4. Diagnostic strategy:
   - For assertion failures: screenshot the actual state, compare to expected, fix assertion or fix app code
   - For timeout errors: check if element exists, check if the app loaded, check selector
   - For navigation errors: verify routes, check base URL, verify dev server is running
5. Fix prioritization: fix the app code if the test correctly describes expected behavior; fix the test if it has wrong expectations
6. When to modify tests vs. when to modify app code (tests reflect SPEC.md requirements; if spec says X, test should expect X, app should deliver X)
7. Re-run workflow: fix one failure category at a time, re-run to verify, repeat until all pass
8. When to accept remaining failures: if a test failure reflects a genuine spec gap, document it rather than force-passing the test

Source all Playwright patterns from RESEARCH.md (Playwright Test Agents docs, accessibility-tree-first execution, init-agents pattern). The Generator follows this pattern as a skill -- it does NOT use `npx playwright init-agents` or spawn actual Playwright agents.
