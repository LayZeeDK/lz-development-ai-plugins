# Phase 02.1: Use Templates for SPEC.md and EVALUATION.md - Research

**Researched:** 2026-03-28
**Domain:** Claude Code plugin architecture -- agent template files, regex-based output parsing, plugin-dev best practices
**Confidence:** HIGH

## Summary

Phase 02.1 extracts the inline document structures from `planner.md` and `evaluator.md` agent definitions into standalone template files that agents reference at runtime. The motivation is reliability: `appdev-cli.mjs` uses regex to extract scores and verdict from EVALUATION.md, so the document format is a contract between the Evaluator agent and the CLI. When the format is defined inline in agent instructions, it drifts -- agents paraphrase, reorder, or omit sections. A canonical template file becomes the single source of truth for both "what the agent produces" and "what the CLI expects to parse."

The plugin-dev plugin's skill-development guidance establishes clear patterns for where supporting files belong. Templates are output-shaping assets (not documentation, not scripts) that agents read at generation time and use as structural blueprints. The `assets/` directory is the plugin-dev-recommended location for "files used within the output Claude produces" -- templates, boilerplate, and structural blueprints fall squarely in this category. However, given the current application-dev plugin's established pattern of using `references/` for agent-consumed files (see `frontend-design-principles.md`), and that templates will be read into context by agents (not blindly copied), placing them under `references/` within the application-dev skill is the more consistent choice.

**Primary recommendation:** Create `SPEC-TEMPLATE.md` and `EVALUATION-TEMPLATE.md` in `plugins/application-dev/skills/application-dev/references/`, then update agent definitions to read these templates via `${CLAUDE_PLUGIN_ROOT}` paths instead of embedding the format inline. Preserve the existing regex contract in `appdev-cli.mjs` unchanged -- the templates must produce output that matches the existing patterns.

## Architecture Patterns

### Recommended File Placement

Based on the plugin-dev plugin's skill-development guidance:

```
plugins/application-dev/
|-- skills/
|   '-- application-dev/
|       '-- references/
|           |-- frontend-design-principles.md    # existing
|           |-- SPEC-TEMPLATE.md                 # NEW
|           '-- EVALUATION-TEMPLATE.md           # NEW
|-- agents/
|   |-- planner.md                               # MODIFIED: references template
|   |-- evaluator.md                             # MODIFIED: references template
|   '-- generator.md                             # UNCHANGED
'-- scripts/
    '-- appdev-cli.mjs                           # UNCHANGED (regex stays same)
```

**Why `references/` not `assets/`:**

The plugin-dev skill-development SKILL.md defines three resource categories:

1. **`references/`** -- "Documentation intended to be loaded into context as needed." Agent reads the file to inform its process and thinking.
2. **`assets/`** -- "Files not intended to be loaded into context, but rather used within the output Claude produces." Files that get copied or modified directly.
3. **`scripts/`** -- Executable code.

Templates are read by agents to inform the structure of their output -- they are context-loaded reference material that shapes behavior, not files that get blindly copied. The existing `frontend-design-principles.md` reference file follows exactly this pattern: the Planner reads it before writing the Visual Design Language section. The templates follow the same read-then-produce pattern: the agent reads the template, then produces a document following that structure.

Additionally, the application-dev plugin already has the `references/` directory populated. Adding files there is consistent with the existing plugin structure.

### Pattern: Agent References External Template

**What:** Agent definition instructs the agent to read a template file before producing output, replacing an inline format specification.

**When to use:** When the document structure is a contract (consumed by another system), or when the format is long enough to bloat the agent definition.

**Current pattern (planner.md, lines 53-141):**
```markdown
## Output Format

Create or overwrite `SPEC.md` in the working directory using this structure:

\```
# <Product Name> -- <Tagline>

## Overview
...
\```
```

**New pattern:**
```markdown
## Output Format

Read the SPEC template at `${CLAUDE_PLUGIN_ROOT}/skills/application-dev/references/SPEC-TEMPLATE.md`.
Create or overwrite `SPEC.md` in the working directory following the template structure exactly.
Do not add, remove, or rename top-level sections -- the template defines the canonical format.
```

**Current pattern (evaluator.md, lines 243-320):**
```markdown
### 8. Write EVALUATION.md

Write your report to `evaluation/round-N/EVALUATION.md`...

Use this exact format:

\```
# Evaluation Report -- <Product Name>
...
\```
```

**New pattern:**
```markdown
### 8. Write EVALUATION.md

Write your report to `evaluation/round-N/EVALUATION.md` where N is the current evaluation round number.
Also save screenshots to `evaluation/round-N/screenshots/`.

Read the evaluation template at `${CLAUDE_PLUGIN_ROOT}/skills/application-dev/references/EVALUATION-TEMPLATE.md`.
Follow the template structure exactly. Do not rename sections or change the Scores table format --
the orchestrator's CLI parses specific patterns from this file.
```

### Anti-Patterns to Avoid

- **Duplicating the format in both template and agent definition.** The agent definition must reference the template, not repeat it. Duplication defeats the single-source-of-truth goal.
- **Making templates too rigid with exact wording.** Templates should define structure (section headings, table format) not exact prose. Agent creativity in content is fine; structural deviation is not.
- **Changing the scores table format in the template without updating the regex.** The regex `\|\s*(Product Depth|Functionality|Visual Design|Code Quality)\s*\|\s*(\d+)\/10` is the invariant contract. The template must produce output matching this pattern.
- **Using `assets/` for templates that need to be read into context.** Assets are for files used in output without loading into context (images, fonts, boilerplate copied as-is). Templates that guide agent behavior belong in `references/`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Template loading | Custom template-loading script | Agent Read tool + `${CLAUDE_PLUGIN_ROOT}` path | Agents already have Read in their tool allowlist; the existing pattern (frontend-design-principles.md) proves this works |
| Template variable substitution | Mustache/Handlebars-style `{{round}}` placeholders | Agent fills in values naturally from prompt context | LLM agents understand structural templates without formal substitution; adding a templating engine adds complexity for zero benefit |

## Common Pitfalls

### Pitfall 1: Agent Ignores Template and Invents Own Format

**What goes wrong:** The agent reads the template but produces output with different section names, missing sections, or reformatted tables.
**Why it happens:** LLMs paraphrase by nature. Without strong enforcement language, the agent may treat the template as a suggestion.
**How to avoid:** Use explicit enforcement phrasing in the agent definition: "Follow the template structure exactly. Do not rename sections or change the table format." The evaluator.md already uses "Use this exact format:" -- preserve that strength of phrasing.
**Warning signs:** `appdev-cli.mjs round-complete` returns an error about missing scores. The regex fails to match.

### Pitfall 2: Template and Agent Instructions Contradict

**What goes wrong:** Template says one thing, remaining inline instructions say something slightly different. The agent gets confused or follows the wrong one.
**Why it happens:** Incomplete migration -- some formatting instructions left inline after template extraction.
**How to avoid:** After extracting the format to a template, remove ALL inline format specifications from the agent definition. Keep only: (1) the instruction to read the template, (2) behavioral guidance (how to think, what mindset to apply), (3) the self-verification checklist. The self-verification items should be framed as "verify your output matches the template" rather than restating the template structure.
**Warning signs:** Agent output has a hybrid structure -- some sections from the template, some from residual inline instructions.

### Pitfall 3: Breaking the Regex Contract

**What goes wrong:** Template author changes the scores table format (e.g., `Score: 8/10` instead of `| 8/10 |`), and the CLI can no longer extract scores.
**Why it happens:** Template feels like "just a markdown file" and someone edits it without considering downstream parsers.
**How to avoid:** Add a comment block at the top of EVALUATION-TEMPLATE.md marking the regex-sensitive sections: "WARNING: The scores table format and verdict heading are parsed by appdev-cli.mjs. Do not change the table column structure or verdict heading format."
**Warning signs:** `round-complete` errors with "Could not extract all 4 scores from report."

### Pitfall 4: Path Resolution Failure

**What goes wrong:** Agent cannot find the template file at runtime because `${CLAUDE_PLUGIN_ROOT}` does not resolve correctly.
**Why it happens:** Unlikely given existing working references (Planner already reads `${CLAUDE_PLUGIN_ROOT}/skills/application-dev/references/frontend-design-principles.md`), but possible if the path is typed incorrectly.
**How to avoid:** Use the exact same path pattern as the existing working reference. Test by running the planner agent once after changes.
**Warning signs:** Agent output says "I could not find the template file" or produces output without template structure.

## Code Examples

### appdev-cli.mjs extractScores() -- The Regex Contract

```javascript
// Source: plugins/application-dev/scripts/appdev-cli.mjs, lines 79-124
function extractScores(reportPath) {
  // ... file reading ...

  const scorePattern = /\|\s*(Product Depth|Functionality|Visual Design|Code Quality)\s*\|\s*(\d+)\/10/gi;
  const scores = {};
  let match;

  while ((match = scorePattern.exec(content)) !== null) {
    const key = match[1].toLowerCase().replace(/\s+/g, "_");
    scores[key] = parseInt(match[2], 10);
  }

  // Expects exactly 4 scores
  if (Object.keys(scores).length !== 4) {
    // ... error ...
  }

  // Also extracts verdict
  const verdictMatch = content.match(/##\s*Verdict:\s*(PASS|FAIL)/);
  const verdict = verdictMatch ? verdictMatch[1] : null;

  return { scores, verdict };
}
```

**Critical patterns the EVALUATION-TEMPLATE.md must preserve:**

1. Scores table row format: `| <Criterion Name> | <N>/10 | ...`
   - Criterion names exactly: `Product Depth`, `Functionality`, `Visual Design`, `Code Quality`
   - Score format exactly: `<integer>/10`
2. Verdict heading format: `## Verdict: PASS` or `## Verdict: FAIL`

## Open Questions

1. **Should the template include prose guidance or just structural skeleton?**
   - What we know: The current inline format in evaluator.md includes both structure (the markdown template) and behavioral guidance (e.g., "Scoring 'Partial' strictly: ..."). These are two different concerns.
   - What's unclear: Whether behavioral guidance should stay inline in the agent definition or move to the template.
   - Recommendation: Keep behavioral guidance in the agent definition. Move only the structural skeleton to the template. The template is the "what the output looks like" contract. The agent instructions are the "how to think" guidance. This separation keeps the template clean for regex-contract purposes and keeps behavioral instructions where the agent naturally reads them.

2. **Should Generator also read the SPEC template?**
   - What we know: The Generator reads SPEC.md (the output of the Planner), not the template. The Generator does not need to know the format -- it needs to know the content.
   - What's unclear: Whether the Generator benefits from knowing the expected structure to better navigate the spec.
   - Recommendation: No. The Generator is a consumer of SPEC.md content, not a producer. Only the Planner needs the template.

3. **Should the self-verification checklists remain inline or reference the template?**
   - What we know: Both planner.md and evaluator.md have "Self-Verification" sections that check whether the output contains required elements (verdict line, scores table, features section, etc.).
   - Recommendation: Keep self-verification inline but simplify it to reference the template: "Verify your output contains all sections defined in the template." The specific items can remain as a quick checklist but should not redefine the format.
