  <name>Task 1: Rename appdev-state.mjs to appdev-cli.mjs and rewrite round-complete with score extraction</name>
  <files>plugins/application-dev/scripts/appdev-cli.mjs</files>
  <action>
1. Use `git mv plugins/application-dev/scripts/appdev-state.mjs plugins/application-dev/scripts/appdev-cli.mjs` to rename the file.

2. Rewrite the `round-complete` subcommand signature. It currently accepts `--round N --verdict PASS|FAIL [--scores json] [--feature-count N]`. Change it to accept `--round N --report <path>` where `<path>` is the path to EVALUATION.md (e.g., `evaluation/round-2/EVALUATION.md`). Remove `--verdict`, `--scores`, and `--feature-count` flags -- the CLI now extracts these from the report file.

3. Add `extractScores(reportPath)` function that:
   - Reads the file at `reportPath` using `readFileSync`
   - If file does not exist, returns `{ error: "Report file not found: <path>" }`
   - Uses regex to extract scores from the markdown table: match `| Product Depth | X/10` etc. for all 4 criteria (Product Depth, Functionality, Visual Design, Code Quality). Use case-insensitive matching with flexible whitespace: `/\|\s*(Product Depth|Functionality|Visual Design|Code Quality)\s*\|\s*(\d+)\/10/gi`
   - Computes `total` as sum of 4 scores
   - Extracts verdict by matching `/##\s*Verdict:\s*(PASS|FAIL)/`
   - If any of the 4 scores or verdict cannot be extracted, returns `{ error: "..." }` with a descriptive message
   - Returns `{ scores: { product_depth, functionality, visual_design, code_quality, total }, verdict }`

4. Add `computeEscalation(rounds)` function implementing the escalation vocabulary per CONTEXT.md decisions:
   - Takes the `rounds` array from state (each entry has `.scores.total`)
   - Round 1 (no previous data): always return `{ level: "E-0", label: "Progressing" }`
   - E-IV Catastrophic: single-round drop >50% of previous total, OR current total <= 5
   - E-III Regression: 2 consecutive total-score declines (current < prev AND prev < prevPrev)
   - E-II Plateau: <=1 point improvement over 3-round window (`current.total - prevPrev.total <= 1`). Requires at least 3 rounds.
   - E-I Decelerating: score improved but delta is smaller than previous delta
   - E-0 Progressing: score improved >1 point
   - Check in priority order: E-IV, E-III, E-II, E-I, E-0. First match wins.
   - Edge cases: delta of exactly 0 or 1 without a 3-round window -> E-I Decelerating. Single decline (not 2 consecutive) -> E-I Decelerating.

5. Add `determineExit(rounds, escalation, maxRounds)` function:
   - If current round's verdict is "PASS": return `{ exit_condition: "PASS", should_continue: false }`
   - If escalation is E-II: return `{ exit_condition: "PLATEAU", should_continue: false }`
   - If escalation is E-III or E-IV: return `{ exit_condition: "REGRESSION", should_continue: false, best_round: N }` where `best_round` is the round with highest total score (use `findBestRound` helper)
   - If current round >= maxRounds (default 10): return `{ exit_condition: "SAFETY_CAP", should_continue: false }`
   - Otherwise: return `{ exit_condition: null, should_continue: true }`

6. Add `findBestRound(rounds)` helper that returns the round entry with the highest `scores.total`.

7. Rewrite `cmdRoundComplete(argv)` to:
   - Parse `--round N` and `--report <path>` (both required)
   - Call `extractScores(reportPath)` -- if error, output the error JSON and exit 1
   - Read current state
   - Create round entry: `{ round, verdict, scores, escalation: null }`
   - Upsert into `state.rounds` (same logic as current: find existing by round number or append)
   - Call `computeEscalation(state.rounds)` to get the escalation for this round
   - Set the round entry's `escalation` field to the computed level (e.g., "E-0")
   - Set the round entry's `escalation_label` field (e.g., "Progressing")
   - Call `determineExit(state.rounds, escalation, 10)` to get exit decision
   - Build trajectory array from all rounds: `state.rounds.map(r => ({ round: r.round, total: r.scores.total, escalation: r.escalation }))`
   - Write updated state
   - Output the combined JSON: `{ round, verdict, scores, escalation, escalation_label, exit_condition, should_continue, best_round (if REGRESSION), trajectory }`

8. Keep all existing subcommands working (init, get, update, complete, delete, exists). The `init` state shape gets an implicit update: round entries now include `escalation` and `escalation_label` fields. The `complete` subcommand remains unchanged. Remove `--feature-count` from the codebase entirely (LOOP-06 deferred to Phase 3).

9. Important: the `--report` flag in `parseArgs` should consume a single token (file path), not multi-token like `--prompt`. Ensure this works correctly with the existing parseArgs logic (it already handles single-token values for non-prompt flags).
