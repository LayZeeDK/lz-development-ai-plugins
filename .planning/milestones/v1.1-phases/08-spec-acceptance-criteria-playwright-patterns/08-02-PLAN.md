Create NEW file PLAYWRIGHT-EVALUATION.md in the evaluator references directory. This is a technique-pure shared reference following the dependency inversion principle -- it describes HOW to interact with the browser efficiently, not WHO uses it or WHAT to look for. Each critic's agent definition will specify which sections to read.

Structure the file with a brief header (purpose + token efficiency rationale) then 7 technique sections:

## eval-first
- What: Structured JSON via `npx playwright-cli eval` -- targeted data extraction.
- Why: eval returns ~100 tokens of targeted data vs snapshot's ~2000 tokens for a full accessibility tree. Use eval when you know what data you need; snapshot when you need to discover element structure.
- Commands: 2-3 examples like `eval "document.title"`, `eval "document.querySelectorAll('img').length"`, `eval "getComputedStyle(document.body).backgroundColor"`.

## write-and-run
- What: 5-step workflow for acceptance test generation that keeps test execution entirely outside agent context.
- Why: Writing tests to disk and running via `npx playwright test --reporter=json` uses ~5 tool calls vs ~30+ interactive browser commands. Test content stays on disk, agent reads only structured JSON results.
- Steps: (1) Read SPEC.md criteria, (2) `npx playwright-cli snapshot` for selector discovery, (3) Write acceptance-tests.spec.ts mapping criteria to test assertions, (4) `npx playwright test <path> --reporter=json` to execute, (5) Read JSON results for pass/fail counts and failure details.
- Include a skeleton acceptance test file showing the criteria-to-test mapping pattern. Use `@playwright/test` imports with `test.describe` per feature, `test()` per criterion. Show accessibility-tree-first selectors (getByRole, getByLabel, getByText) from snapshot ref IDs. Include a comment above each test citing the SPEC criterion it verifies. Show 2-3 test examples mapping different criterion types (count assertion, user action, error state).

## snapshot-as-fallback
- What: `npx playwright-cli snapshot` saves accessibility tree to disk for element/ref discovery.
- Why: Only use when you need element ref IDs for interaction or when eval alone cannot determine element structure. Snapshot output is verbose (~2000 tokens) -- prefer eval for targeted queries.
- Commands: `npx playwright-cli snapshot` (saves to .playwright-cli/page-{timestamp}.yml).

## resize+eval
- What: Viewport resize then eval for responsive checks.
- Why: Checking responsive behavior via eval after resize is more token-efficient than taking screenshots at every breakpoint.
- Commands: Show the resize-eval-screenshot sequence. Use the same command name as the existing perceptual-critic.md (`viewport` -- match existing agent definitions for consistency). Example: `npx playwright-cli viewport 320 800`, then `eval "document.querySelectorAll('.card').length"`, then `screenshot --filename=mobile-320.png` at key viewpoints only.

## console filtering
- What: `npx playwright-cli console error` returns only error-level messages.
- Why: Unfiltered `console` returns all messages (info, warn, debug, error) which fills context with noise. Error-only filtering captures functional and visual issues without token waste.
- Commands: `npx playwright-cli console error` (filtered -- use this), NOT `npx playwright-cli console` (unfiltered -- avoid).

## test healing (evaluation context)
- What: Re-snapshot, update selectors, re-run when acceptance tests fail on selectors.
- Key insight: Selector failures = test maintenance (UI renamed/moved elements). Assertion failures = real findings (behavior changed). This distinction matters -- selector failures should be healed, assertion failures should be reported.
- Flow: (1) Identify failing tests, (2) categorize as selector timeout vs assertion failure, (3) for selector timeouts: re-snapshot, find updated selectors, update test file, re-run. (4) for assertion failures: report as findings.
- Boundary: Critics heal their OWN tests only. Critics never modify application source code. The Generator's heal loop (Plan -> Generate -> Heal skill) fixes the app; the critic's heal loop fixes only test selectors.

## round 2+ test reuse
- What: Reuse/heal/regenerate decision tree for acceptance tests in rounds 2 and later.
- Flow: (1) Copy acceptance-tests.spec.ts from evaluation/round-{N-1}/projection/ to evaluation/round-N/projection/. (2) Run copied tests. (3) Read results and decide:
  - Reuse when: all tests pass OR tests fail on assertions only (behavior changed, structure intact -- report assertion failures as findings)
  - Heal when: 1-2 selector timeouts (minor UI rename/move -- re-snapshot, update selectors, re-run)
  - Regenerate when: multiple selector timeouts OR >50% tests fail on timeouts (major UI restructure -- full write-and-run from SPEC.md + fresh snapshot)
- No new tests in rounds 2+. Round 1 writes tests covering all SPEC.md criteria. Consistent tests across rounds enable meaningful CLI trajectory analysis.
- Missing prior tests fallback: If round N-1 has no acceptance-tests.spec.ts (crash or skip), fall back to round 1 behavior (write fresh tests from SPEC.md).
- Each round directory is self-contained. Prior rounds are immutable after completion.

Style guidelines for the entire file:
- WHY-based rationale throughout (not ALL-CAPS emphasis). Consistent with project conventions.
- No "Used by" annotations. This is technique-pure -- consumers declare which sections they read.
- No specific token counts (they vary by page complexity).
- No Playwright JSON reporter format documentation (Playwright does not follow semver -- critics read actual output).
- playwright-cli commands use `npx playwright-cli` prefix consistently.
- Target ~120-150 lines total. Enough depth for actionable guidance, not so much that it overwhelms a 60K token context budget.
