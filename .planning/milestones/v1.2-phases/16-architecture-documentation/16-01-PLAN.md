**3. Turing Test Framing (NEW section)**
- Critics as interrogators: each critic interacts with the running application the way a user would (navigating, clicking, submitting forms, resizing, stressing). The critic cannot distinguish a "good" application from a "real" application except through observable behavior.
- Product surface as evaluation boundary: critics see only what a user would see (rendered pages, console output, network responses). Source code, build configuration, and implementation choices are behind the barrier.
- The Turing test is asymmetric: the Generator knows everything (spec, code, build), but critics know only the spec and the running product. This asymmetry is the adversarial pressure that drives quality.
- Connection to Ashby's black box methodology: the critic observes what the application DOES, not what the Generator says it does. Observable behavior is the only valid evaluation signal.

**4. Agent Roles and Separation (update existing)**
- Update from 3 roles (Planner, Generator, Evaluator) to 5 roles organized in two tiers:
  - Planning and Generation tier: Planner (specification), Generator (implementation)
  - Critic Ensemble tier: multiple specialized critics, each scoring a different quality dimension
- For each critic, describe its evaluation APPROACH in principle terms (not methodology details):
  - Perceptual critic: evaluates visual quality and cross-page consistency through screenshots and visual inspection
  - Projection critic: evaluates functional correctness through automated behavioral tests (write-and-run pattern)
  - Perturbation critic: evaluates resilience through adversarial stress testing (beyond-spec conditions)
- Keep the Orchestrator description ("dumb coordinator" principle) but update to reflect N-critic spawning
- Keep the two-layer enforcement model section (tool allowlists + prompt guards) -- this is a durable principle

**5. Cybernetics Principles (update existing)**
- Keep the escalation vocabulary table (E-0 through E-IV) but describe it as a principle of named operating regimes, not specific trigger values
- Update the damping principle section (already good, keep it)
- Update score-based convergence detection to include:
  - EMA smoothing principle: exponentially weighted moving average dampens noise in score trajectory, preventing false convergence/regression signals from single-round anomalies
  - Scaled thresholds principle: all convergence thresholds are derived from the scoring range (number of dimensions times max score), not hardcoded. This means adding or removing dimensions automatically recalibrates convergence detection.
  - Dual-path signal architecture principle: safety-critical decisions (catastrophic failure, pass) use raw scores for immediate response; trend decisions (plateau, regression, progress) use smoothed scores for noise resistance. Hybrid decisions use both.
  - Schmitt trigger hysteresis principle: asymmetric thresholds for entering vs. exiting convergence states prevent oscillation around a boundary (a larger improvement is needed to signal "progressing" than the stall needed to signal "plateau")
- Add requisite variety principle (Ashby's Law): the evaluation system must have at least as much variety as the system being evaluated. A monolithic evaluator lacks variety to assess visual design, functional correctness, and adversarial resilience simultaneously. The ensemble architecture provides requisite variety through critic specialization.
- Add feedback loop topology: Generator -> Application -> Critics -> Scores -> CLI Aggregator -> Orchestrator -> Generator. The loop is mediated by files (not in-memory state), which means any component can be restarted without losing the feedback signal.
